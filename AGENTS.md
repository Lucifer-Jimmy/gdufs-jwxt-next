# GDUFS JWXT Next 开发指南

本文档是本仓库所有开发工作的首要约束。它记录已确认的产品需求、技术边界、目录职责和开发规范。实现前应先阅读本文档；当需求、架构、上游接口或部署方式发生变化时，必须在同一变更中同步维护本文档。

## 1. 项目基本信息

- 项目名称：GDUFS JWXT Next
- 项目类型：前后端分离的 Web 应用
- 目标用户：广东外语外贸大学学生
- 产品定位：在学校统一认证和教务系统之上，提供更清晰、顺手的个人教务数据查询与学业分析体验
- 默认语言：简体中文（`zh-CN` / `zh-Hans`）
- 部署目标：Cloudflare Workers
- 参考项目：`gdufs-jwxt-next-origin/`
- 免责声明：本项目由个人开发维护，不代表广东外语外贸大学或学校教务部门，也不替代官方成绩单

## 2. 已确认需求

以下要求为硬性约束，未经用户明确确认不得更改：

1. 前端和后端均使用 TypeScript，业务源代码中不得新增 JavaScript 或其他语言实现。
2. 前后端分离，根目录必须以 `frontend/` 和 `backend/` 明确区分职责。
3. 前端以 shadcn/ui 作为主要 UI 组件库和实现体系。页面、表单、反馈、导航、数据展示和交互控件应优先由 shadcn/ui 组件组合实现，再按项目需求在其 token、variant 和组件源码层扩展；不得另建一套平行的主组件库。
4. 整体应用必须能够部署到 Cloudflare Workers，后端不得依赖 Node.js 专属运行时能力、常驻进程、本地文件系统或传统服务器会话。
5. 应用不得持久化任何用户数据。所有教务数据从上游接口实时获取、即时使用。
6. 登录、MFA、SSO、个人信息、成绩查询和成绩详情等后端业务行为，以 `gdufs-jwxt-next-origin/` 的生产实现为迁移依据。
7. 不得参考、继承或复刻 `gdufs-jwxt-next-origin/` 的 `PRODUCT.md`、`DESIGN.md`、前端样式、模板结构、视觉实现、品牌描述或信息架构。旧前端只可用于核对与后端相关的业务功能、字段含义和必要用户流程。
8. 所有前端设计和实现任务必须使用 `impeccable` skill 辅助，并遵守本文档的前端体验规范。`impeccable` 用于建立新项目自己的设计方向、审查和打磨体验；具体组件实现仍以 shadcn/ui 为主。
9. 项目已于 2026-07-18 获得用户授权进入正式开发阶段；实现必须遵循第 11.2 节的阶段顺序和各阶段验收门槛。
10. 前端框架确定为 React + Vite，后端框架确定为 Hono + Cloudflare Workers。
11. 生产环境采用单 Worker 同源部署：同一源下由 Worker 提供前端静态资源，并由 Hono 处理 `/api/*` 请求。
12. 无服务端持久化会话：认证状态使用加密、完整性受保护的 `HttpOnly Cookie` 由浏览器直接携带；个人信息和成绩只保留在当前页面内存，不使用 `sessionStorage`。
13. 严格限流使用 SQLite-backed Cloudflare Durable Objects，仅保存不可逆主体摘要、动作、计数和时间等自动过期的安全元数据；不得保存任何用户业务数据或认证状态。
14. 首个版本必须完整提供学业概览、GPA/学分统计、分学期趋势图、成绩筛选、单科成绩详情以及 Excel/CSV 导出。这里只迁移业务能力，不继承旧前端的信息架构或视觉实现。
15. 毕业学分与通识学分进度按专业匹配多套静态规则。规则表由项目维护、版本化并随前端发布，不通过数据库或运行时存储管理。
16. 同一专业默认跨年级使用相同培养方案；只有学校资料或用户明确说明存在差异时才按年级或培养方案版本细分，规则结构需保留这些未来扩展维度。
17. 支持最新版及前两个主要版本的 Chrome、Edge、Firefox 和 Safari，并适配 iOS Safari 与 Android Chrome。微信内置浏览器尽力兼容，但不作为阻断发布的强制验收目标。

## 3. 项目边界

### 3.1 目标能力

首个版本必须覆盖以下主流程与用户能力：

- 账号密码登录学校统一认证系统。
- 获取并展示脱敏手机号。
- 发送手机 MFA 验证码，并遵守上游返回的重发等待时间。
- 校验 MFA 验证码，获取 SSO ticket 并换取教务系统 Cookie。
- 获取学生个人信息，包括学号、姓名、学院和专业。
- 实时获取全量成绩。
- 提供学业概览，展示基于当前实时成绩计算的关键学业指标。
- 按学期和课程属性筛选成绩。
- 查看单门课程的成绩组成。
- 在前端计算 GPA、学分、毕业学分进度、通识修读进度和分学期趋势。
- 以可访问的趋势图展示分学期 GPA 等趋势，并提供等价文本或数据表达。
- 在浏览器端导出 Excel/CSV，不将导出内容上传或暂存在后端。
- 正确处理登录过期、上游超时、上游页面结构变化、请求过频和解析失败。

除非用户明确扩展范围，不增加选课、评教、考试安排修改或任何会改变教务系统数据的功能。本项目默认只读。

### 3.2 旧项目的使用规则

优先参考以下生产代码：

- `gdufs-jwxt-next-origin/apps/authserver/services.py`：统一认证、密码加密、MFA 和 ticket 获取。
- `gdufs-jwxt-next-origin/apps/jwxt/services.py`：教务 Cookie、个人信息、成绩列表和成绩详情。
- `gdufs-jwxt-next-origin/apps/accounts/session.py`：多阶段认证状态、过期策略和刷新约束。
- `gdufs-jwxt-next-origin/apps/accounts/views.py` 与 `apps/jwxt/views.py`：业务错误到 HTTP/用户提示的映射。
- `gdufs-jwxt-next-origin/tests/`：既有业务行为和边界条件。

不得把 `prototypes/` 当作正式实现依据。旧项目的 `PRODUCT.md` 与 `DESIGN.md` 均不属于新项目参考资料；不得从中继承产品定位、用户画像、信息架构、品牌个性、设计原则、颜色、字体、组件或交互方案。也不得参考 `templates/`、`static/css/` 和 `static/js/` 来设计新前端。新项目的产品和设计文档必须根据用户当前需求重新访谈、确认并独立编写。

迁移不是逐行翻译 Python。必须先识别协议、请求顺序、Cookie 变化、HTML/JSON 解析规则和异常语义，再用 Worker 兼容的 Web API 与 TypeScript 重新实现。

## 4. 技术栈与架构基线

以下为已确认的技术与架构基线。任何会改变这些基线的后续需求均按第 12 节的确认边界处理。

### 4.1 仓库结构

```text
frontend/                 独立前端应用
  src/
    components/           通用组件与 shadcn/ui 组件
    features/             按业务能力组织的页面逻辑
    routes/               页面或路由入口
    lib/                  API 客户端、计算与通用工具
    styles/               全局样式和设计 token
  tests/                  前端测试

backend/                  独立 Cloudflare Worker API
  src/
    routes/               HTTP 路由与输入/输出适配
    services/             authserver、jwxt 等上游业务服务
    parsers/              上游 HTML/JSON 解析
    session/              无持久化加密状态封装
    schemas/              请求、响应和上游数据校验
    errors/               领域错误与 HTTP 错误映射
    security/             加密、日志脱敏、安全响应头
  tests/                  后端单元、契约和集成测试

gdufs-jwxt-next-origin/   只读迁移参考，不属于新应用构建与部署产物
docs/                     详细开发文档、协议说明、运维手册与验证记录
AGENTS.md                 本开发指南
PRODUCT.md                新项目独立产品定位与体验原则
DESIGN.md                 新项目 shadcn/ui 主导的种子视觉系统
```

可在实施时按所选框架补充文件，但不得打破 `frontend/` 与 `backend/` 的边界。前后端如需共享 API 类型，可通过生成类型或明确的 workspace 包实现；不得让前端直接导入后端运行时代码。

### 4.2 前端

- 语言：TypeScript，开启严格类型检查。
- 应用框架：React + Vite。
- UI：shadcn/ui + Tailwind CSS。shadcn/ui 是前端的主要组件库，不仅用于少量基础控件。
- 图标：优先使用 shadcn/ui 默认生态中的 Lucide 图标。
- 数据校验：所有后端响应在运行时校验，不因 TypeScript 类型而信任网络数据。
- 数据获取：使用统一 API 客户端，集中处理超时、取消、认证失效和错误映射。
- 计算职责：GPA、学分统计、筛选、排序、图表派生数据和导出优先在前端基于当前响应计算。
- 学分规则：在前端使用版本化、只读的静态规则表，以标准化专业标识作为默认匹配维度，并在数据结构中保留年级或培养方案版本维度。
- 浏览器端不得永久保存成绩、个人信息、认证 Cookie、密码或验证码。

#### 专业学分规则表

- 每条规则必须包含稳定规则 ID、专业匹配值、规则版本、生效范围、毕业总学分要求、通识学分要求及计算所需的课程分类条件。
- 规则来源必须可追溯到学校发布的培养方案或用户确认资料，并记录来源说明与核对日期；不得根据旧前端显示结果反推或猜测规则。
- 没有特殊说明或明确资料时，同一专业的所有年级使用同一套规则，不得仅因入学年份不同擅自拆分培养方案。
- 上游专业名称在匹配前可做明确、可测试的规范化和别名映射，但不得使用模糊匹配把用户分配到不确定规则。
- 若未来有明确资料表明同一专业存在不同年级或培养方案，规则表应通过预留维度细分；若当前上游接口无法唯一识别适用规则，必须停止展示毕业/通识完成比例，并提示“该专业规则暂无法准确匹配”，仍可展示不依赖培养方案的已修学分和 GPA。
- 未配置专业、规则冲突、字段缺失或规则版本无法确定时必须安全降级，不得回退到全校统一默认学分值，也不得展示可能错误的“已达毕业要求”。
- 规则变更必须包含针对匹配、边界学分、课程分类和降级行为的单元测试，并在同一改动中记录规则版本变化。

### 4.3 后端

- 语言：TypeScript，开启严格类型检查。
- 运行时：Cloudflare Workers 标准运行时。
- HTTP 层：Hono，运行在 Cloudflare Workers 标准运行时中。
- 上游请求：使用标准 `fetch`、`Headers`、`Request`、`Response`、`URL` 和 Web Crypto API。
- HTML 解析：使用明确支持 Cloudflare Workers 的结构化解析能力，不以正则表达式解析一般 HTML。对于旧接口中嵌入脚本的固定数据片段，应隔离解析器并用真实脱敏 fixture 测试。
- 数据校验：对客户端输入、上游 JSON 和解析结果进行运行时 schema 校验。
- 不得依赖 Node.js `fs`、`net`、`tls`、原生二进制扩展、进程内全局 Session 或服务器本地磁盘。

### 4.4 Cloudflare 部署

- 使用单个 Cloudflare Worker 同源部署前后端。源码、依赖和构建职责仍严格保持 `frontend/` 与 `backend/` 分离，不因部署单元合并而混合模块边界。
- `/api/*` 请求由 Hono 后端处理；其他页面路由与构建后的前端文件由 Cloudflare Workers Static Assets 提供。
- 前端调用相对路径 `/api/...`，生产环境不配置独立 API 域名。
- 不使用 D1、KV、R2、Queues、数据库或外部持久化服务保存用户数据或认证状态。Durable Objects 仅作为严格限流的匿名元数据存储，适用范围不得扩展。
- 密钥只通过 Cloudflare Secrets 提供，不写入仓库、客户端 bundle 或普通环境变量示例值。
- Worker 必须设置明确的上游请求超时、错误处理和 `Cache-Control: no-store`。
- 前后端同源，因此默认不启用跨域 CORS。认证 Cookie、Origin/Referer 校验和 CSRF 防护均以该同源拓扑设计；若未来改为分域部署，必须先重新进行安全设计并更新本文档。
- 构建和部署配置必须排除 `gdufs-jwxt-next-origin/`。

### 4.5 工程工具与依赖基线

- 包管理器使用 `pnpm`，根目录采用 workspace 管理 `frontend/` 与 `backend/`。只提交根目录 `pnpm-lock.yaml`，不得混用 npm、Yarn 或 Bun 锁文件。
- Node.js 使用当前活跃 LTS 版本，并通过仓库配置固定最低版本；本地、CI 和 Cloudflare 构建环境必须保持一致。
- 前端路由使用 React Router；服务端数据请求与失效管理使用 TanStack Query，但查询结果只存在内存，不启用持久化插件。
- 前后端运行时 schema 使用 Zod。共享 API 契约应放在不含运行时副作用的独立 workspace 包或由 OpenAPI 生成；前端不得导入 Worker 路由或服务实现。
- 图表优先使用 Recharts；Excel 导出使用 SheetJS，CSV 使用项目内纯函数生成。专项库必须服从 shadcn/ui token、可访问性和包体积约束。
- 单元与集成测试使用 Vitest；React 组件测试使用 Testing Library；端到端和视觉验证使用 Playwright。
- 代码质量使用 ESLint 与 Prettier。TypeScript、lint、测试、构建和 Wrangler 配置必须提供根目录统一命令。
- 新增依赖必须说明用途，不允许引入与 shadcn/ui 重叠的综合 UI 库，也不允许引入依赖 Node.js 原生 API 而无法在 Workers 运行的后端包。

## 5. 无存储与会话设计

“不存储任何用户业务数据”按以下严格基线执行。唯一持久化例外是第 6.4 节定义的匿名、限时安全限流元数据：

- 禁止使用服务端数据库、KV、对象存储或持久卷保存用户业务数据、认证状态或上游响应。Durable Objects 不得用于限流以外的用途。
- 禁止把用户数据、成绩、密码、验证码、SSO ticket 或上游 Cookie写入日志、分析平台、错误追踪上下文或缓存。
- 请求处理期间可以在内存中短暂持有完成当前请求所需的数据；响应结束后不保留。
- 多阶段认证所需的临时状态和登录后的教务 Cookie，必须封装为限时、加密且完整性受保护的 `HttpOnly` Cookie，由浏览器随请求带回。Worker 不保存对应副本。
- MFA 临时认证 Cookie 默认有效期为 10 分钟，供用户完成验证码接收与输入；不得滑动续期。验证码或上游认证会话更早过期时，以更早的时间为准。
- 正式登录 Cookie 默认采用 2 小时无操作过期。仅在成功处理已认证 API 请求后滑动续期；静态资源访问、未认证请求和失败请求不得延长会话。
- 正式登录 Cookie 从首次签发起最长有效 8 小时，滑动续期不得突破该绝对上限。上游教务 Cookie 更早失效时，应立即清除本地登录态并要求重新登录。
- Cookie 应使用 `Secure`、`HttpOnly`、合理的 `SameSite`、最小 `Path`，并支持密钥轮换。Cookie 内容不得包含密码或 MFA 验证码。
- Cookie 体积必须受控；只携带恢复上游会话所需的最小字段，并处理浏览器约 4 KiB 限制。
- 退出、认证失效或上游 Cookie 过期时，后端必须通过过期 `Set-Cookie` 清除客户端认证状态。
- 响应默认 `Cache-Control: no-store, private`，不得让 Cloudflare Cache 或浏览器 HTTP cache 保存个人数据。
- 前端个人信息、成绩和派生统计仅保留在当前页面内存；页面刷新后重新从上游实时查询。禁止使用 `sessionStorage`、`localStorage`、IndexedDB 或 Cache API 保存这些数据和任何认证信息。

该方案是无服务端持久化，不等于无状态数据传递。任何改变这一边界的实现必须先更新本文档并获得用户确认。

## 6. 后端业务与 API 规范

### 6.1 认证流程

实现必须保留旧项目的协议顺序和安全语义：

1. 请求统一认证登录页，解析 `pwdEncryptSalt` 和 `execution`。
2. 使用与上游兼容的 AES-CBC 规则加密密码，仅在当前请求生命周期持有明文密码。
3. 提交账号密码并验证是否进入 MFA 流程。
4. 获取脱敏手机号和必要的临时认证 Cookie。
5. 发送 MFA 验证码并遵守上游 `codeTime`。
6. 校验验证码并获取 SSO ticket。
7. 使用 ticket 换取教务 Cookie，并立即验证个人信息接口以确认登录有效。
8. 将最小登录态以短期加密客户端状态返回，不持久化账号、个人信息或 Cookie。

账号、密码、验证码、ticket、完整 Cookie 和个人数据不得出现在错误消息或日志中。日志仅记录错误类别、请求 ID、上游阶段和脱敏后的必要诊断信息。

### 6.2 教务查询

- 每次查询直接请求上游教务系统。
- 全量成绩默认沿用旧项目单页上限 `300`，但需把该值集中配置并对达到上限的情况发出不含个人数据的告警。
- 成绩详情的四个标识参数必须来自同一条成绩记录；必须校验字段存在、长度和格式。
- 上游返回非 JSON、登录页 HTML 或缺少预期字段时，区分 Cookie 过期与页面结构变化。
- 后端只做必要的协议适配和数据规范化，不保存成绩，也不承担可由前端完成的展示筛选和统计缓存。

### 6.3 API 契约

- 首版 API 固定使用 `/api/v1` 前缀。破坏性变更必须发布新版本；不得在同一路径下静默改变字段语义。
- 首版路由固定如下：

| 方法   | 路径                      | 用途                                   | 认证要求                          |
| ------ | ------------------------- | -------------------------------------- | --------------------------------- |
| `GET`  | `/api/v1/health`          | Worker 健康检查，不访问学校上游        | 无                                |
| `POST` | `/api/v1/auth/login`      | 校验账号密码并创建 MFA 临时状态        | 无，严格限流                      |
| `GET`  | `/api/v1/auth/mfa`        | 获取脱敏手机号、发送状态与剩余等待时间 | MFA 临时 Cookie                   |
| `POST` | `/api/v1/auth/mfa/send`   | 发送 MFA 验证码                        | MFA 临时 Cookie，严格限流         |
| `POST` | `/api/v1/auth/mfa/verify` | 校验验证码并建立正式登录态             | MFA 临时 Cookie，严格限流         |
| `POST` | `/api/v1/auth/logout`     | 清除 MFA 与正式登录 Cookie             | 可选正式认证，始终幂等            |
| `GET`  | `/api/v1/me`              | 从上游实时获取个人信息                 | 正式登录 Cookie                   |
| `GET`  | `/api/v1/grades`          | 从上游实时获取全量成绩                 | 正式登录 Cookie                   |
| `POST` | `/api/v1/grades/refresh`  | 用户主动刷新全量成绩                   | 正式登录 Cookie，账号级 30 秒限流 |
| `POST` | `/api/v1/grades/detail`   | 实时获取单科成绩组成                   | 正式登录 Cookie                   |

- 成绩详情使用 `POST` JSON body，避免课程内部标识进入 URL、浏览器历史、代理日志或分析系统；该接口仍为只读操作。
- 请求和响应均定义显式 TypeScript 类型与运行时 schema。
- 成功响应直接返回有名资源对象，不额外套无意义的 `data` 层。错误响应固定为 `{ "error": { "code": string, "message": string, "requestId": string, "retryAfterSeconds"?: number } }`。
- 不把原始上游 HTML、堆栈、内部 URL、Cookie 或异常文本直接返回前端。
- 合理使用 HTTP 状态码：输入错误 `400`、未认证/过期 `401`、频率限制 `429`、上游失败 `502`、上游超时 `504`。
- 所有 `429` 响应同时返回标准 `Retry-After` header 与 `retryAfterSeconds`；前端倒计时以服务端值为准。
- 修改认证状态的 `POST` 请求必须验证 `Content-Type: application/json` 和同源 `Origin`；不接受表单编码作为应用 API 输入。
- `/health` 之外的 API 响应一律使用 `Cache-Control: no-store, private`。所有响应携带或透传不含敏感信息的请求 ID。
- 前端不得依赖上游系统的原始字段命名；后端应通过边界类型隔离上游协议，必要的课程详情标识除外。

### 6.4 限流说明

严格限流使用 SQLite-backed Cloudflare Durable Objects，并遵守以下约束：

- 初始使用 16 个固定分片。对限流主体计算带服务端密钥的 HMAC，再根据摘要稳定映射到其中一个分片；不得在 Durable Objects 中保存真实账号、学号或原始 IP。
- 分片只是存储与并发边界，不是共享额度。同一分片中的不同主体使用各自的 `subject_hash + action` 唯一键，独立计算窗口、计数与冷却时间。
- 每次限流检查和额度登记必须在同一 Durable Object 内原子完成，确保同一主体的并发请求不能同时绕过限制。
- Durable Object 只完成轻量检查与写入，随后立即返回 Worker。访问统一认证或教务系统等慢速上游请求不得在 Durable Object 内执行。
- 限流记录只包含 `subject_hash`、`action`、窗口或最后请求时间、计数与过期时间。记录必须按用途设置最短合理保留期并自动清理，最长保留不超过 24 小时，除非后续安全需求另行确认。
- 成绩手动刷新按账号 HMAC 严格限制为每 30 秒一次。MFA 验证码发送同时遵守本地严格限流和学校认证系统返回的等待时间；两者取更严格者。
- 登录与 MFA 防滥用采用账号 HMAC 和 IP HMAC 等独立维度组合判断，默认阈值如下。任一维度触发即拒绝；成功登录只清理该账号的连续失败计数，不清理 IP 窗口：

| 动作               | 主体             | 默认限制                            | 记录保留                |
| ------------------ | ---------------- | ----------------------------------- | ----------------------- |
| 账号密码登录提交   | 账号 HMAC        | 10 分钟最多 5 次                    | 10 分钟                 |
| 账号密码登录提交   | IP HMAC          | 10 分钟最多 30 次                   | 10 分钟                 |
| MFA 验证码发送     | 账号 HMAC        | 10 分钟最多 3 次，24 小时最多 10 次 | 最长 24 小时            |
| MFA 验证码发送     | IP HMAC          | 10 分钟最多 20 次                   | 10 分钟                 |
| MFA 验证码校验失败 | MFA 流程 ID HMAC | 单次流程最多 5 次                   | 随 10 分钟 MFA 状态过期 |
| 成绩手动刷新       | 账号 HMAC        | 每 30 秒最多 1 次                   | 30 秒                   |

- MFA 发送还必须遵守学校返回的 `codeTime`，本地窗口与上游冷却取更严格者。MFA 校验达到 5 次失败后必须清除临时认证 Cookie，要求重新进行账号密码登录。
- 限流按被 Worker 接受并准备执行的尝试计数；上游超时或失败不得返还额度，避免攻击者利用失败响应绕过限制。
- 分片数量和映射算法必须版本化。不得直接把 16 改为其他数量，以免同一主体迁移到新分片并在过渡期绕过限流；扩容必须设计兼容迁移方案。
- Durable Objects 故障或限流状态不可用时，成绩刷新、验证码发送和登录尝试等受保护动作必须默认拒绝并返回可恢复错误，不得失效放行。
- 配置 Cloudflare 用量监控与告警，关注请求数、行读写、存储与错误率，防止异常流量耗尽免费额度。

## 7. 前端产品与设计规范

本项目是产品型工具界面，设计服务于查询任务，不做营销落地页。每次设计、重构、审查或视觉验收都必须使用 `impeccable` skill。新项目自己的产品上下文位于根目录 `PRODUCT.md`，种子视觉系统位于根目录 `DESIGN.md`；实现必须以两者和本文档为依据。禁止复制、改写或默认沿用旧项目 `PRODUCT.md` 与 `DESIGN.md` 的内容。

### 7.1 核心体验

- 登录后应让用户快速回答：当前 GPA 是多少、已修多少学分、离目标还差多少、各学期趋势如何。
- 概览与逐门成绩核对保持清晰的信息架构，不把所有内容堆在单页。
- 成绩属于正式、敏感数据；视觉应高级、优雅、丝滑，品质来自精确层级、克制品牌色、自然动效和及时反馈，而非夸张装饰。
- 不复刻官方教务系统的拥挤表格、灰蓝陈旧感和僵硬控件，也不走通用 SaaS 营销模板风格。
- 首屏必须是可使用的登录或应用界面，不制作无必要的 landing page。

### 7.2 shadcn/ui 使用

- shadcn/ui 是页面 UI 的主要实现来源。新增组件前先检查 shadcn/ui 是否已有对应组件或推荐组合；存在时应优先采用，而不是自行从零实现或引入另一套 UI 库。
- 优先复用和组合 shadcn/ui 的标准交互语义，包括 Button、Input、Form、Alert、Dialog、Sheet、Tabs、Table、Select、Dropdown Menu、Tooltip、Skeleton 和 Toast/Sonner。
- 可以直接维护 shadcn/ui 生成到仓库中的组件源码，并通过统一 token、variant 和组合组件适配本项目；定制必须保留其可访问性和标准交互语义。
- 除非 shadcn/ui 确实不覆盖某项专业能力且用户确认，不引入会与其重叠的综合 UI 组件库。图表、表格增强等专项库也必须包裹在本项目的 shadcn/ui 视觉与交互体系内。
- 使用熟悉图标表达纯工具操作，并提供可访问名称和必要 tooltip；不要用带文字的圆角矩形替代明确的刷新、下载、关闭等图标操作。
- 组件样式统一通过设计 token 和 variant 管理，禁止页面内散落魔法颜色、任意圆角和重复 class 片段。
- 不嵌套卡片，不把每个区块都做成悬浮卡片。卡片仅用于真正独立、重复或需要边界的内容，圆角不超过 8px，除非后续设计系统明确规定。
- 标准控件保持标准行为，不为追求“特别”重造滚动条、下拉框、弹窗或表单语义。

### 7.3 视觉与排版

- 产品界面默认采用克制的中性色层级与单一主强调色；强调色只用于主操作、当前选择和状态提示。
- 颜色使用 OKLCH token，并验证 WCAG 2.1 AA 对比度：正文至少 `4.5:1`，大号或加粗文本至少 `3:1`。
- 产品 UI 使用一致、清晰的无衬线字体体系；标题采用固定字号层级，不用随视口缩放的超大展示字。
- 成绩数值应便于纵向核对，采用右对齐和适合的等宽数字特性。
- 长段说明控制在约 `65–75ch`；表格可更宽，但必须提供结构化的移动端适配。
- 禁止渐变文字、装饰性玻璃拟态、装饰性网格背景、渐变光球、重复等大卡片网格、侧边粗色条和模板化小号大写 eyebrow。
- 不使用由单一紫色、蓝紫、米色、深蓝或棕橙色系统治整个界面的单调配色。

### 7.4 状态与交互

- 所有交互组件必须覆盖默认、hover、focus-visible、active、disabled、loading 和 error 状态。
- 内容加载优先使用保持布局稳定的 Skeleton；不得用居中 spinner 代替整块内容结构。
- 空状态要说明原因并给出下一步；错误状态要保留上下文和可恢复操作。
- 登录、验证码发送、验证码倒计时、成绩刷新、筛选、详情和导出都要提供即时且准确的反馈。
- 动效只用于解释状态变化，常规时长控制在 `150–250ms`，使用自然的 ease-out，不做编排式页面入场。
- 所有动效必须适配 `prefers-reduced-motion`。
- 弹窗必须管理焦点、支持 `Esc` 关闭，并避免在可内联展开或渐进披露时滥用 modal。
- 布局必须有稳定尺寸约束，动态内容、加载态和 hover 不得引起无意义位移。

### 7.5 响应式与可访问性

- 支持手机和桌面浏览器；响应式通过结构变化实现，例如导航折叠、列收缩和移动端成绩条目视图，而不是简单缩小字体。
- 强制兼容最新版及前两个主要版本的 Chrome、Edge、Firefox 和 Safari，以及对应平台仍受支持的 iOS Safari 与 Android Chrome。
- 微信内置浏览器作为尽力兼容目标：应完成主流程冒烟验证并修复成本合理的兼容问题，但其特有内核差异不得阻断发布。若关键安全 API 或浏览器能力不可用，应显示清晰的不支持提示，不得降低加密或认证安全要求换取兼容。
- 所有功能必须可通过键盘完成，焦点顺序与视觉顺序一致，焦点样式清晰可见。
- 表单必须有可感知标签、字段级错误、提交级错误和正确的自动填充属性。
- 图表不是数据的唯一表达方式，颜色不是状态的唯一线索；必须提供文本或表格等价信息。
- 页面使用正确标题层级、landmark、表头关联和 ARIA 语义；优先使用原生 HTML 语义。
- 必须检查最长中文文案、错误文案、窄屏和缩放场景，任何文本不得溢出或遮挡其他内容。

## 8. 安全与隐私规范

- 这是处理账号凭据和学业数据的敏感应用，安全优先于便利性和视觉效果。
- 只允许向明确白名单中的学校认证和教务域名发起后端请求；绝不接受客户端提供任意 ticket URL 或上游 URL，防止 SSRF。
- 对重定向地址逐步验证协议、主机和路径，不盲目跟随跨域重定向。
- 密码与验证码只存在于完成对应请求所需的最短时间内，不写 Cookie、不写前端状态持久层、不记录日志。
- 加密状态使用 Web Crypto 的现代 AEAD 算法；旧系统要求的 AES-CBC 仅限与统一认证密码协议兼容的那一步，不用于本应用会话封装。
- 状态 Cookie 必须具备版本、签发时间、最后活动时间、绝对过期时间和用途绑定，避免跨流程重放，并确保滑动续期不能突破绝对上限。
- 同源部署时对改变认证状态的请求实施 Origin/Referer 校验；分域部署时另行确定 CSRF 方案。
- 设置最小权限的 CSP、`Referrer-Policy`、`X-Content-Type-Options`、`Permissions-Policy` 等安全响应头。
- 前端不得加载会接触个人数据的第三方分析、广告或会话回放脚本。
- 测试 fixture 必须彻底脱敏，不提交真实学号、姓名、手机号、成绩、Cookie、ticket 或凭据。

### 8.1 Cookie 容量与拆分约束

- 开发认证业务前必须使用结构等价的脱敏 fixture 测量 MFA 状态和正式登录状态在加密、认证与 Base64 编码后的实际字节数。
- 单个 Cookie 的完整 `Set-Cookie` 值必须控制在 3,800 字节以内，为浏览器属性和未来字段预留空间；所有应用 Cookie 的请求头总量目标不超过 6 KiB。
- MFA 临时状态与正式登录状态必须使用不同 Cookie 名称、用途绑定和加密上下文。完成 MFA 后立即清除临时 Cookie。
- 如果单个状态超过容量，不得改用 `sessionStorage`、KV、数据库或 Durable Objects 保存认证状态。应先最小化字段，再按固定编号拆分为少量 `HttpOnly` Cookie，并在密文载荷中包含整体版本、分片总数和完整性校验。
- 缺少任一 Cookie 分片、顺序异常、版本不符或解密失败时，必须清除整组状态并要求重新登录，不得尝试部分恢复。
- 测试必须覆盖浏览器 Cookie 上限、代理 header 上限、分片丢失、旧版本 Cookie 和密钥轮换场景。

### 8.2 配置与密钥

- 至少定义以下 Worker secrets：会话 AEAD 主密钥、限流 HMAC 密钥。两者必须独立，不得复用。
- 非敏感配置至少包括：上游请求超时、MFA 状态有效期、登录无操作与绝对过期时间、限流阈值、分片版本、成绩页大小和允许的前端源。
- secrets 只通过 `wrangler secret` 或 Cloudflare 控制面注入；本地使用不提交的 `.dev.vars`。仓库只提供不含真实值的 `.dev.vars.example` 或等价说明。
- 新增任何本地凭据文件、预览环境令牌、证书、私钥或可能包含敏感响应的调试输出时，必须同时检查并维护根目录 `.gitignore`。任意目录下的 `.dev.vars` 及其环境后缀文件必须默认忽略，只允许提交不含真实值的 `.dev.vars.example`。
- 密钥必须具备版本号并支持轮换。解密可在有限窗口接受当前与上一版本，签发只能使用当前版本；过渡结束后删除旧版本。

## 9. TypeScript 与代码规范

- 前后端分别拥有明确的 `tsconfig`，启用 `strict`，并优先启用 `noUncheckedIndexedAccess`、`exactOptionalPropertyTypes` 等严格选项。
- 禁止无说明的 `any`、非空断言和类型强制转换。边界数据先校验，再收窄类型。
- 使用小而明确的领域类型，不直接让上游松散对象贯穿系统。
- 业务服务不依赖具体框架的 Request/Response；路由层负责 HTTP 适配，服务层负责协议与领域行为。
- 解析、加密、日志脱敏、错误映射和统计计算应保持纯函数或可独立测试。
- 错误使用有判别字段的领域错误表达，不依赖字符串匹配控制流程。
- 命名与代码使用英文；用户可见文案和产品文档使用简体中文。
- 注释解释协议原因、上游怪异行为和安全约束，不重复代码表面含义。
- 不提交生成产物、密钥、真实响应数据或本地环境文件。
- 优先使用仓库已有工具与模式；新增依赖前检查 Worker 兼容性、包体积、维护状态和许可证。

## 10. 测试与验收

### 10.0 开发启动验证

以下验证必须在完整业务开发之前完成，并记录脱敏结果：

1. **Worker 上游连通性**：在 Cloudflare 远程预览或非生产 Worker 中分别请求统一认证登录页和教务入口，验证 DNS、TLS、重定向、响应编码、Cookie header 与超时行为。不得使用真实账号完成自动化登录。
2. **Cookie 容量**：使用结构等价的脱敏上游 Cookie fixture 验证 MFA 与正式登录载荷加密后的大小、分片策略和浏览器往返行为。
3. **运行时兼容性**：验证 AES-CBC 上游兼容逻辑、会话 AEAD、HMAC、HTML 解析器、Zod、Hono 和 Durable Objects SQLite 均能在 `wrangler dev` 的 Workers 运行时执行。
4. **单 Worker 路由**：验证 `/api/v1/*` 始终进入 Hono，React 客户端路由与静态资源由 Workers Static Assets 正确回退，API 404 不得回退成前端 HTML。

若 Cloudflare 出口被学校上游阻断、必要响应 header 无法取得或认证状态无法在合理 Cookie 预算内传递，必须停止完整开发并重新确认架构，不能以本地开发服务器可用代替生产可行性。

### 10.1 后端

- 为认证每个阶段编写单元测试和基于脱敏 fixture 的解析测试。
- 覆盖成功、账号密码错误、MFA 错误/过期、验证码重发限制、ticket 缺失、Cookie 过期、上游超时、非 JSON、字段缺失和页面结构变化。
- 使用请求 mock 验证 URL、方法、关键 header、表单字段、重定向策略和 Cookie 传递，不在常规测试中访问真实学校系统。
- 加密状态测试覆盖篡改、MFA 过期、2 小时无操作过期、8 小时绝对过期、滑动续期边界、用途不匹配、密钥轮换和超出 Cookie 大小限制。
- API 契约测试验证状态码、schema、安全 header 和响应不含敏感字段。

### 10.2 前端

- 单元测试覆盖 GPA、学分、趋势、筛选、排序、导出映射和错误转换。
- 组件测试覆盖表单校验、加载/空/错误状态、认证失效、刷新倒计时和键盘操作。

- 图表必须验证非空、尺寸稳定，并在无 Canvas/SVG 可视信息时仍有可访问的数据替代。

### 10.3 完成定义

一项功能只有在以下条件全部满足时才算完成：

- 行为符合本文档和已确认需求。
- 类型检查、lint、单元测试和相关端到端测试通过。
- Cloudflare Workers 本地运行环境验证通过。
- 敏感信息、缓存和日志边界经过检查。
- 前端通过 `impeccable` 辅助审查以及手机/桌面视觉验收。
- 实现、API、环境变量、业务流程、架构、部署或验证结果发生变化时，`docs/` 中的相关详细开发文档已在同一变更中同步更新。

### 10.4 专业规则验收

- 学分规则框架、schema、匹配器和安全降级可以在规则资料不完整时开发。
- 某个专业只有在培养方案来源、毕业总学分、通识要求、课程分类条件与核对日期齐全，并通过测试后，才能向用户展示毕业/通识完成比例。
- 未验证专业必须明确显示“该专业规则暂未配置或无法准确匹配”，不得使用旧项目固定值、相近专业规则或全校平均值代替。
- 静态规则表应包含覆盖情况清单，发布说明必须列出本版本已验证和未验证的专业范围。

## 11. 开发流程

1. 开始任务前阅读 `AGENTS.md` 和相关旧项目生产代码。
2. 若需求会改变数据边界、认证方案、API 契约、技术栈或部署方式，先向用户确认并更新本文档。
3. 先定义边界类型、运行时 schema 和可测试的业务行为，再实现路由和界面。
4. 后端迁移以旧实现和旧测试为事实依据，但须适配 Worker 的无状态模型。
5. 前端不得查看旧 `PRODUCT.md`、`DESIGN.md` 或旧视觉实现来寻找设计方案；先通过用户访谈与当前需求独立建立新项目上下文，再用 `impeccable` 辅助设计决策，并以 shadcn/ui 为主要组件体系实现。
6. 保持改动聚焦，不顺带重构无关区域，不覆盖用户已有改动。
7. 开发过程中及时维护 `docs/` 中的相关文档；实现与文档必须保持一致，不得把文档补写长期留作后续任务。
8. 提交或暂存前检查 `git status --short --ignored` 和待提交差异，确认 `.gitignore` 已覆盖新增的本地敏感文件，且暂存区、文档、fixture、日志和构建产物中不含凭据或用户数据。
9. 提交前执行与改动风险相称的自动化与视觉验证。

### 11.1 开发文档维护

- `docs/` 是项目详细开发文档的统一目录，应记录足以让新开发者理解、运行、验证、排障和继续开发本项目的具体信息；`AGENTS.md` 继续负责最高层约束和已确认边界，两者不得相互冲突。
- `docs/` 至少应按实际开发进度覆盖：总体架构与目录职责、API 契约、认证与上游协议、Cookie 会话和安全设计、Durable Objects 限流、前端数据流与计算规则、环境配置、Cloudflare 构建部署、测试策略、验证记录、故障排查和关键技术决策。
- 新增或修改功能时，必须检查对应文档是否需要同步更新。凡是改变接口字段、请求顺序、状态流转、安全边界、配置项、部署步骤、运行命令、依赖约束或已知限制的改动，都必须在同一变更中维护相关 `docs/` 文档。
- 开发验证应记录环境、日期、脱敏结果、结论和未完成项。不得在文档中写入真实账号、姓名、学号、手机号、成绩、Cookie、ticket、密钥、完整上游响应或其他敏感信息。
- 文档应使用简体中文，命令、路径、类型名和协议字段保留其原始英文名称。内容必须具体、可执行并与当前代码一致，不能只写概括性进度或复制代码表面行为。
- 如果某项实现尚未完成或结论尚未验证，文档必须明确标注状态、适用范围和阻塞项，不得把计划描述成已完成能力。

### 11.2 建议实施阶段

1. **阶段 0：可行性验证**。建立最小 workspace 与 Wrangler 配置，只实现开发启动验证所需探针、Cookie 容量测试和 Worker 运行时兼容测试。
2. **阶段 1：契约与安全基础**。定义 `/api/v1` schema、统一错误、请求 ID、加密 Cookie、Origin 校验、日志脱敏和 Durable Objects 限流。
3. **阶段 2：认证与上游适配**。迁移账号密码、MFA、SSO、个人信息、成绩和详情解析，以旧后端生产代码与脱敏 fixture 为事实依据。
4. **阶段 3：前端基础**。初始化 React、Vite、Tailwind CSS 和 shadcn/ui，落地真实 OKLCH token、字体、动效、应用壳与所有基础状态；随后运行 `$impeccable document` 更新 `DESIGN.md` 并生成 sidecar。
5. **阶段 4：用户功能**。实现概览、GPA/学分、趋势、筛选、详情和导出，并接入专业规则安全降级。
6. **阶段 5：生产加固**。执行端到端、视觉、无障碍、浏览器兼容、限流并发、Cookie 轮换和 Cloudflare 预览验证。

不得为了展示前端进度跳过阶段 0–2，也不得在真实 token 尚未落地前把 seed `DESIGN.md` 当作最终设计规格。每个阶段完成时必须同步维护 `docs/` 中的阶段说明、验证记录和后续阻塞项。
